Code Exchange Database & Backend

Rol: Backend / Database Developer Opleiding: HBO-ICT Propedeuse Blok: Blok 2 Jaar: 2023-2024 Samenwerking: Duo project (2 personen) Technologieën: SQL (phpMyAdmin), HTML/CSS, TypeScript (basis)

Context van het Project

Het doel van dit project was het bouwen van een interactief platform genaamd Code Exchange, geïnspireerd op Stack Overflow. Dit platform moest gebruikers in staat stellen om code snippets te delen, vragen te stellen en antwoorden te geven binnen een gemeenschap van developers.

Het platform moest gebruikersgegevens opslaan, relaties tussen data correct verwerken, en informatie efficiënt kunnen ophalen uit een relationele database. Het was een duo project waarbij we volgens Scrum sprints werkten, zoals voorgeschreven in Blok 2 van de Propedeuse HBO-ICT opleiding.

Mijn focus lag op de backend en database: het ontwerpen van de datastructuur, het schrijven van SQL queries, en het zorgen dat data correct wordt opgehaald en verwerkt. Ik heb mij bewust niet primair gefocust op frontend ontwikkeling, maar op dataverwerking en backend logica.

Mijn Concrete Bijdrage

Focus op Database & Backend

Mijn primaire verantwoordelijkheid lag bij het ontwerpen en gebruiken van de database, het schrijven van SQL queries, en het correct koppelen van tabellen met INNER JOINs. Dit was mijn eerste diepgaande ervaring met relationele databases en SQL, en ik heb hier intensief aan gewerkt.

Ik heb mij bewust niet gefocust op frontend ontwikkeling, maar op dataverwerking en logica. Dit betekende dat ik nauw samenwerkte met mijn duo partner om ervoor te zorgen dat de database structuur aansloot bij de frontend vereisten, maar mijn eigen werk concentreerde zich op de backend kant van de applicatie.

SQL & Dataverwerking (zeer concreet)

Ik heb uitgebreid gewerkt met SQL en verschillende typen queries geschreven:

  • SELECT queries: Voor het ophalen van data uit de database. Ik schreef queries om gebruikersinformatie, code snippets, vragen en antwoorden op te halen.
  • INSERT queries: Voor het toevoegen van nieuwe records aan de database. Ik implementeerde queries om nieuwe gebruikers, code snippets en interacties op te slaan.
  • UPDATE queries: Voor het bewerken van bestaande records. Ik zorgde ervoor dat gebruikers hun profiel en content konden updaten.
  • DELETE queries: Voor het verwijderen van records. Ik implementeerde logica om content te kunnen verwijderen wanneer nodig.

Intensief werken met INNER JOINs: Een belangrijk onderdeel van mijn werk was het gebruik van INNER JOIN queries om data uit meerdere tabellen te combineren. Dit was essentieel voor het ophalen van gebruikersinformatie gekoppeld aan hun content, of het tonen van vragen met bijbehorende antwoorden en gebruikersgegevens.

Query optimalisatie: Ik heb queries geoptimaliseerd om gebruikersinformatie correct te koppelen aan content en overzichtelijke datasets terug te geven aan de applicatie. Dit betekende dat ik nagedacht heb over welke data wanneer nodig was en hoe ik queries kon structureren voor efficiëntie.

Voorbeelden van use cases die ik heb geïmplementeerd:

  • Ophalen van gebruikers en bijbehorende content via joins: Ik schreef queries die gebruikersinformatie combineerden met hun geplaatste code snippets, vragen en antwoorden. Dit vereiste INNER JOINs tussen gebruikers, content en interactietabellen.
  • Opslaan van nieuwe records in meerdere gerelateerde tabellen: Bij het plaatsen van een code snippet moest informatie worden opgeslagen in meerdere tabellen (content, tags, relaties). Ik zorgde ervoor dat deze transacties correct werden afgehandeld.
  • Filteren en structureren van data voor weergave: Ik schreef queries die data filterden op criteria (bijvoorbeeld alle code snippets van een specifieke gebruiker, of alle vragen met een bepaald aantal antwoorden) en deze structureerden voor weergave in de applicatie.
  • Bewerken en verwijderen van bestaande records via SQL logica: Ik implementeerde queries om bestaande records te updaten of verwijderen, waarbij ik rekening hield met relaties en integriteit van de database.

Database ontwerp

Ik heb gewerkt met een relationele database beheerd via phpMyAdmin. Dit was mijn eerste ervaring met het ontwerpen van een complete database structuur voor een applicatie.

Nadenken over tabelstructuur: Ik heb nagedacht over welke tabellen nodig waren (gebruikers, content, tags, relaties) en welke kolommen elk tabel nodig had. Ik leerde over verschillende datatypes en wanneer je welke gebruikt.

Relaties tussen entiteiten: Ik heb relaties gelegd tussen verschillende entiteiten: gebruikers hebben content, content heeft tags, gebruikers kunnen op content reageren. Dit vereiste foreign keys en een goed begrip van hoe entiteiten met elkaar verbonden zijn.

Normalisatie van data: Ik leerde over normalisatie en waarom het belangrijk is om data niet te dupliceren maar te normaliseren. Dit betekende dat ik data in logische tabellen opdeelde en relaties legde waar nodig.

Begrip van impact op performance en schaalbaarheid: Door dit project ontwikkelde ik begrip van hoe backend keuzes invloed hebben op performance, schaalbaarheid en onderhoudbaarheid. Een goed ontworpen database zorgt voor snellere queries en makkelijker onderhoud, terwijl een slecht ontworpen database problemen kan veroorzaken op de lange termijn.

TypeScript / Logica (beperkt, maar benoemd)

Mijn bijdrage aan TypeScript was beperkt tot het ondersteunen van backend-logica. Ik gebruikte TypeScript voor eenvoudige dataverwerking en de koppeling tussen database resultaten en applicatielogica.

Dit betekende geen complexe frontend interacties, maar functionele ondersteuning van dataflows. Ik schreef code die database resultaten verwerkte, transformeerde voor gebruik in de applicatie, en zorgde ervoor dat data correct van en naar de database ging.

Samenwerking & Proces

Ik werkte in een duo samenwerking waarbij taken duidelijk verdeeld waren. Mijn verantwoordelijkheid lag bij het database gedeelte binnen de Scrum sprints, terwijl mijn partner zich focuste op de frontend ontwikkeling.

Afstemmen van backend werk met frontend vereisten: Ik leerde hoe belangrijk het is om regelmatig te communiceren over welke data nodig is voor de frontend. Ik moest vragen stellen zoals: "Welke informatie heb je nodig?" en "Hoe wil je de data gestructureerd hebben?" Dit hielp me om queries te schrijven die precies aansloten bij wat de applicatie nodig had.

Communiceren over datamodellen en query structuren: Ik leerde hoe belangrijk het is om je werk uit te leggen aan anderen. Ik communiceerde over hoe de database was opgezet, welke relaties er waren, en hoe queries werkten. Dit zorgde ervoor dat mijn partner begreep hoe de data gestructureerd was en hoe hij/zij deze kon gebruiken.

Door in een duo te werken volgens Scrum sprints leerde ik ook over planning, het verdelen van taken, en het behalen van sprint doelen. Dit was een waardevolle ervaring in teamwerk en projectmanagement.

Wat ik heb Geleerd in dit Project

Dit project was mijn eerste diepgaande ervaring met SQL. Ik leerde hoe essentieel SQL is voor het werken met relationele databases en hoe belangrijk het is om goede queries te kunnen schrijven. SELECT, INSERT, UPDATE en DELETE werden concrete tools die ik dagelijks gebruikte.

INNER JOINs essentieel voor realistische applicaties: Ik leerde dat INNER JOINs essentieel zijn voor realistische applicaties. Zonder JOINs kan je alleen data uit één tabel ophalen, maar in werkelijkheid heb je bijna altijd data uit meerdere gerelateerde tabellen nodig. Het begrijpen en correct gebruiken van JOINs was een grote stap vooruit in mijn ontwikkeling als backend developer.

Backend denken: data eerst, UI daarna: Ik ontwikkelde begrip van backend denken: data eerst, UI daarna. Dit betekende dat ik leerde om te denken in termen van datastructuren, relaties en query's, in plaats van eerst te denken over hoe iets eruit ziet. Het begrijpen van de data laag is fundamenteler dan het begrijpen van de presentatie laag.

Database keuzes hebben directe invloed op de rest van de applicatie: Ik leerde hoe database keuzes directe invloed hebben op de rest van de applicatie. Een goed ontworpen database maakt het makkelijker om queries te schrijven, data op te halen en de applicatie te onderhouden. Een slecht ontworpen database zorgt voor problemen op de lange termijn. Dit begrip van de impact van database ontwerp was een belangrijke leerervaring.

Solide basis voor latere backend en full stack projecten: Dit project legde een solide basis voor latere backend- en full stack projecten. Het gaf me de fundamentele vaardigheden in SQL, database ontwerp en backend logica die ik later nodig zou hebben voor complexere projecten. Het was een belangrijke eerste stap in mijn ontwikkeling als software engineer.

Resultaat

Het resultaat was een werkend Code Exchange platform met een goed ontworpen database en functionerende backend logica. Gebruikers konden accounts aanmaken, code snippets delen, vragen stellen en antwoorden geven, waarbij alle data correct werd opgeslagen en opgehaald uit de relationele database.

Door dit project heb ik een solide basis gelegd in SQL en database ontwerp. Ik leerde hoe belangrijk het is om goed na te denken over datastructuren, hoe je efficiënte queries schrijft, en hoe backend keuzes invloed hebben op de hele applicatie. Deze ervaring heeft me geholpen om te begrijpen dat backend development fundamenteler is dan frontend development: zonder een goede database en logica, werkt geen enkele applicatie goed.

Dit project was een belangrijke stap in mijn ontwikkeling als software engineer en heeft me de vaardigheden gegeven die ik nodig had voor latere, complexere projecten. Het liet me zien dat ik graag werk aan de backend kant van applicaties, waar logica en data centraal staan.
