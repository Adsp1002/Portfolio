Typing Game Prototype (LIT)

Rol: Front-end Developer (Web Components) Opleiding: HBO-ICT Propedeuse Blok: Blok 3 Jaar: 2023-2024 Technologie: LIT, JavaScript / TypeScript, HTML, CSS Type: Individueel / teamproject (prototype)

Context van het Project

Het doel van dit project was om een interactieve typing game te bouwen die lijkt op Monkeytype. Deze game moest real time invoer verwerken, statistieken tonen zoals snelheid en nauwkeurigheid, en volledig draaien in de browser.

Dit project diende als introductie tot component based development met LIT. Het was mijn eerste kennismaking met het LIT framework en web components, en ik leerde hoe je moderne front-end architecture gebruikt om interactieve applicaties te bouwen.

Het eindproduct is een playable demo die duidelijk lijkt op Monkeytype: gebruikers kunnen woorden typen die op het scherm verschijnen, krijgen real time feedback op hun typing gedrag, en zien hun statistieken zoals Words Per Minute (WPM) en accuracy percentage.

Mijn Concrete Bijdrage

LIT & Web Components

Dit was mijn eerste project waarbij ik met LIT web components werkte. Ik heb zelf meerdere componenten gebouwd die de kern van de typing game vormden.

Componenten die ik heb gebouwd:

  • Hoofdcomponent voor de game flow: Ik bouwde een hoofdcomponent die de game logica beheerde, de state bijhield, en de verschillende fases van de game coördineerde (start, spelen, einde).
  • Component voor woordweergave: Ik ontwikkelde een component die woorden weergaf en visuele feedback gaf op typing. Dit component hield bij welke woorden correct waren getypt en welke fouten waren gemaakt.
  • Statistieken component voor feedback: Ik bouwde een component dat real-time statistieken toonde zoals WPM, accuracy percentage, en aantal getypte woorden. Dit component update automatisch wanneer de state veranderde.

Gebruik van reactive properties: Ik heb gebruik gemaakt van reactive update de UI automatisch. Dit maakte state management veel eenvoudiger en voorspelbaarder dan met traditionele JavaScript.

Events voor communicatie tussen componenten: Ik gebruikte events voor communicatie tussen componenten. Wanneer de gebruiker een woord correct typte, werd een event gedispatched dat andere componenten konden luisteren, zoals de statistieken-component die dan de WPM en accuracy bijwerkte.

Automatische re-renders bij state wijzigingen: Een van de krachtigste aspecten van LIT die ik leerde was de automatische re-rendering. Wanneer reactive properties veranderden, renderde het component automatisch opnieuw, zonder dat ik handmatig de DOM hoefde te manipuleren. Dit maakte de code veel eenvoudiger en onderhoudbaarder.

Geen complexe routing: Ik gebruikte geen complexe routing; de focus lag op één interactieve game flow. De game had één scherm waar alle actie plaatsvond: het typen van woorden, het tonen van statistieken, en het geven van feedback. Dit hielp me om me te focussen op de game logica en component structuur in plaats van routing complexiteit.

Game logica

Ik heb de volledige game logica gebouwd die de typing game werkend maakte. Dit omvatte timer logica, statistiek berekeningen, en input verwerking.

Logica voor timer: Ik bouwde logica voor het bijhouden van tijd. De game gebruikte een countdown timer die de speeltijd beperkte, of een timer die bijhield hoe lang de gebruiker al aan het typen was. Ik gebruikte JavaScript timing functies om deze timer te beheren en te updaten.

Typing speed berekenen (WPM): Ik implementeerde logica voor het berekenen van typing speed in Words Per Minute (WPM). Dit vereiste het bijhouden van het aantal correct getypte woorden, de tijd die daarvoor nodig was, en het berekenen van de gemiddelde snelheid. Ik update deze berekening real time terwijl de gebruiker typte.

Fouten tellen en accuracy berekenen: Ik bouwde logica voor het tellen van fouten en het berekenen van accuracy percentage. Ik hield bij hoeveel karakters correct waren getypt versus hoeveel fouten waren gemaakt, en berekende hieruit het accuracy percentage. Dit gaf gebruikers directe feedback op hun prestaties.

Toetsenbordinvoer verwerken via key events: Ik implementeerde event handlers voor keyboard events. Wanneer de gebruiker een toets indrukte, controleerde mijn code of dit de juiste toets was voor het huidige woord, of een fout. Ik gebruikte JavaScript keyboard events om deze input te verwerken en te reageren.

Actief feedback geven aan de gebruiker: Ik implementeerde visuele feedback mechanismen:

  • Huidige woorden highlighten: Ik markeerde het woord dat de gebruiker op dat moment moest typen, zodat duidelijk was waar ze waren in de woordenlijst.
  • Foutieve invoer visueel markeren: Wanneer een gebruiker een verkeerde toets indrukte, markeerde ik dit visueel (bijvoorbeeld met rood) zodat de gebruiker direct kon zien dat er een fout was gemaakt.
  • Correcte invoer markeren: Wanneer een woord correct was getypt, markeerde ik dit (bijvoorbeeld met groen) zodat de gebruiker feedback kreeg op hun voortgang.

Woorden selecteren uit woordenlijst: Ik bouwde logica voor het selecteren van woorden uit een vooraf ingestelde woordenlijst, met eenvoudige randomisatie. De woordenlijst bevatte honderden woorden, en de game selecteerde willekeurig woorden uit deze lijst om te tonen. Dit zorgde voor variatie en uitdaging.

UI & Styling

Ik heb de layout opgebouwd met HTML en CSS. De focus lag op functionele duidelijkheid, niet op uitgebreide visuele effecten. Het doel was om een duidelijke, werkende typing game interface te maken.

Feedback zichtbaar maken: Ik maakte feedback zichtbaar via:

  • Kleurveranderingen: Ik gebruikte kleuren om feedback te geven: groen voor correct, rood voor fouten, en andere kleuren om de huidige positie te markeren.
  • Highlights: Ik gebruikte CSS highlights om bepaalde woorden of karakters te benadrukken, zoals het huidige woord dat getypt moest worden.
  • Duidelijke statische UI elementen: Ik creëerde duidelijke UI-elementen voor statistieken (WPM, accuracy), timer, en woordenlijst. Deze elementen waren statisch op de pagina gepositioneerd voor maximale duidelijkheid.

Layout structuur: Ik organiseerde de interface in logische secties: een groot gebied voor de woordenlijst, een paneel voor statistieken, en een timer display. Deze structuur was gebaseerd op Monkeytype en zorgde voor een duidelijke, gebruiks vriendelijke interface.

Resultaat

Het eindresultaat was een playable demo die duidelijk leek op Monkeytype. De game was volledig speelbaar in de browser: gebruikers konden direct beginnen met typen, woorden werden getoond die ze moesten typen, en ze kregen real time feedback op hun typing gedrag.

Functionaliteit: De game functioneerde volledig: woorden werden willekeurig geselecteerd uit een woordenlijst, gebruikers konden deze typen en kregen directe visuele feedback, en statistieken werden real time bijgewerkt. De game hield bij hoeveel woorden correct waren getypt, hoeveel fouten waren gemaakt, en berekende WPM en accuracy.

Gebruikerservaring: Gebruikers kregen direct feedback op hun typing gedrag: ze zagen welke woorden correct waren (groen), welke fouten waren gemaakt (rood), en hun real time statistieken. Dit maakte de game interactief en responsief, precies zoals Monkeytype.

Het prototype toonde dat ik in staat was om een werkende, interactieve game te bouwen met LIT web components, en dat ik begrip had van game logica en UI feedback. Het was een waardevolle eerste ervaring met component based development en moderne front-end architecture.

Wat ik heb Geleerd

Dit project was mijn eerste ervaring met component based frontend development. Ik leerde hoe je applicaties opdeelt in herbruikbare, modulaire componenten, en hoe componenten met elkaar communiceren via events en reactive properties.

State management in LIT: Ik ontwikkelde begrip van state management in LIT. Ik leerde hoe je reactive properties gebruikt om component state te beheren, en hoe state changes automatisch de UI updaten. Dit was een fundamentele verschuiving in hoe ik dacht over front-end development.

Reactivity: Ik leerde wat reactivity betekent: wanneer data verandert, update de UI automatisch. Dit maakte het werken met dynamische data veel eenvoudiger dan met traditionele JavaScript, waarbij je handmatig de DOM moet manipuleren bij elke verandering.

Event gedreven UI's: Ik ontwikkelde begrip van event gedreven UI's. Ik leerde hoe je events gebruikt voor communicatie tussen componenten, en hoe user actions (zoals keyboard input) triggers zijn voor events die componenten kunnen verwerken.

Game logica koppelen aan UI feedback: Ik leerde hoe je game logica koppelt aan UI feedback. Het was niet genoeg om alleen de logica te hebben; de gebruiker moest ook visuele feedback krijgen. Ik leerde hoe je state changes vertaalt naar visuele updates, en hoe je real time feedback geeft op user actions.

Stevige basis voor latere projecten: Dit project legde een stevige basis voor latere projecten met LIT en andere frameworks. Het gaf me de fundamentele vaardigheden in component based development, state management, en event handling die ik nodig had voor complexere projecten. Het toonde me hoe krachtig moderne front-end frameworks zijn voor het bouwen van interactieve applicaties.
